01 What & Why

The Student Hub is a new course marketplace where students discover, purchase, and consume educational content published by providers on Candlefox's existing Marketing Hub. It is architected as a fully independent system — separate application, database, authentication, and deployment — that receives content from the Marketing Hub via webhooks.

Core User Journey

  Browse Catalog ──▶ Search / Filter ──▶ View Course ──▶ Purchase ──▶ Learn ──▶ Review
  (anonymous)        (anonymous)         (anonymous)     (auth req)   (enrolled) (enrolled)

Why a Separate System?

  • Different users — Students and providers have different identity requirements, traffic patterns, and feature needs
  • Independent scaling — Read-heavy catalog browsing (90% of traffic) must not compete with Marketing Hub's write-heavy content management
  • Deployment isolation — Ship student features without risking Marketing Hub stability
  • Technology fit — Edge-first SSG/ISR for catalog pages, separate auth (Clerk vs Supabase Auth), dedicated search index

System architecture: all managed services, edge-first reads, webhook-based integration

Data Flow Patterns

  • Catalog reads (90% of traffic): Edge CDN → ISR cache → (on miss) Server Component → Redis cache → PostgreSQL. Sub-100ms for cache hits
  • Search: Client → Meilisearch (direct, sub-50ms). Index synced from PostgreSQL on content webhook
  • Purchases: GraphQL mutation → Stripe Checkout Session → redirect to Stripe → webhook → enrollment creation in PostgreSQL
  • Content sync: Marketing Hub webhook → verify HMAC → upsert to PostgreSQL + Meilisearch + bust Redis/ISR cache

05 API & Data Layer

GraphQL API (BFF)

A single /api/graphql endpoint serves all client data needs. The schema is built code-first with Pothos, meaning types are defined in TypeScript and the GraphQL schema is derived — no SDL files to keep in sync.

The request pipeline processes every GraphQL operation through: (1) Yoga plugin: auth context from Clerk session, (2) Yoga plugin: Upstash rate limiting, (3) Pothos auth scopes per field/type, (4) Zod input validation, (5) DataLoader batching for N+1 prevention, (6) Drizzle ORM queries to Neon Postgres.

Database Schema

The database stores a read-optimized catalog mirror (synced from Marketing Hub), plus student-owned data: enrollments, progress, reviews, and notifications. Key tables:

TablePurposeKey Relationships
catalog_coursesCourse data mirror from Marketing HubReferenced by enrollments, reviews, search index
catalog_providersProvider profile mirrorOne-to-many with courses
enrollmentsStudent-course access recordsLinks student (Clerk ID) to course; includes Stripe payment ref
progressPer-module completion trackingLinks enrollment to content modules
reviewsStudent ratings and text reviewsOne per student-course pair (unique constraint)

Caching Strategy

Four-layer cache hierarchy: Browser (Cache-Control headers) → Vercel CDN (ISR, 60s TTL) → Upstash Redis (GraphQL response fragments) → Neon PostgreSQL (source of truth). Cache invalidation is event-driven via webhooks: content change → bust Redis keys + revalidatePath() for ISR.

06 Authentication & Security

Authentication (Clerk)

Clerk manages all student identity — registration, login, social OAuth (Google, Apple), MFA, and session management. It is completely separate from the Marketing Hub's Supabase Auth. Auth is enforced at three independent layers:

  1. Edge middleware — Vercel edge function verifies JWT in <1ms, redirects unauthenticated users on protected routes (/dashboard/*, /checkout/*)
  2. Pothos auth scopes — GraphQL resolvers declare $authenticated or $public scopes; unauthenticated access to protected fields returns UNAUTHENTICATED error
  3. Service-layer ownership — Every data query includes WHERE student_id = ctx.userId to prevent cross-student data access (IDOR protection)

GitHub Actions runs quality gates (Biome lint ~10s, tsc ~15s, Vitest ~30s, integration tests ~45s, schema validation ~5s) in parallel with Vercel preview deployments (~60–90s). CI must pass before PR merge. Production deploys run Drizzle migrations first, then atomic app deployment with zero downtime.

Observability

  • Logs: Pino structured JSON → Vercel Log Drain. Every request includes requestId, userId, operation name, duration
  • Errors: Sentry with stack traces, breadcrumbs, and session replay. Alerts on new error types and spikes
  • Performance: Vercel Analytics for Core Web Vitals (LCP, CLS, INP) per route. Alert if LCP >2.5s on catalog pages

08 Scalability

The architecture scales from 10K to 100K+ MAU without code changes — only service tier upgrades. The read-heavy catalog workload (90% of traffic) is absorbed entirely by caching layers (ISR edge cache + Redis), leaving the database to handle only writes and cache misses.

09 Key Decisions & Trade-offs

DecisionTrade-off Accepted
Separate system from Marketing HubData duplication (catalog mirror) for independent deployability and scaling
Webhook sync (not API polling)Eventual consistency (~seconds) for decoupled systems and no polling overhead
GraphQL over RESTSchema complexity for flexible queries, reduced over-fetching, and type-safe codegen
ISR over full SSR for catalogUp to 60s staleness for near-zero server cost and global sub-100ms loads
All managed servicesLess infrastructure control for zero DevOps overhead and ~$150/mo at launch
Separate auth (Clerk vs Supabase)No SSO between student/provider for complete system isolation
Stripe Checkout (redirect)Brief domain redirect for PCI SAQ-A (lightest compliance tier)
No message queue at launchLess async resilience for reduced complexity. NATS can be added later
Architecture Principles

(1) Edge-first reads — ISR + CDN absorb 90% of traffic at zero server cost. (2) Managed everything — No self-hosted infrastructure; ~$150/mo at launch. (3) Defense in depth — Auth at edge, API, and service layer independently. (4) Type-safe pipeline — Drizzle → Pothos → urql → React (zero type gaps). (5) Externalize sensitive ops — Auth in Clerk, payments in Stripe, no credentials or card data in our database.
