01 Executive Summary

The Student Hub is a new consumer-facing web application that serves as a courses and skills marketplace — a platform where students discover, purchase, and consume short-form upskilling courses (ranging from a few hours to a few weeks). Think of it as a purpose-built Udemy-style marketplace, tightly integrated with our existing Provider Marketing Hub where learning institutions already publish and manage their course catalogs.

While the Provider Marketing Hub serves hundreds to a few thousand institutional users and is built on React, TypeScript, GraphQL, and Supabase, the Student Hub must be architected for a fundamentally different scale: millions of concurrent student users. This difference in scale — along with different usage patterns (high-read catalog browsing, bursty purchase traffic, interactive content consumption) — justifies a purpose-built architecture rather than extending the existing provider-side stack.

Key Architectural Decisions

  • Separate system, shared course data — Student profiles, enrollments, progress, and transactions live in a dedicated database. Course catalog data is synced from the existing Supabase instance (read-only from the Student Hub's perspective).
  • Edge-first, SSR/SSG hybrid frontend — The catalog (the most-visited surface) is statically generated and served from the edge for sub-second load times worldwide. Personalized pages use server-side rendering.
  • Separate authentication system — Students authenticate through a dedicated auth provider, decoupled from the Supabase Auth used by providers.
  • Stripe-powered commerce — All financial transactions route through Stripe, keeping payment data off our servers entirely (PCI SAQ-A compliance).
  • English-first, i18n-ready — Launching in English with the architecture wired for future localization without rewrites.
Relationship to Provider Marketing Hub

The Student Hub is a consumer of course data published through the Provider Marketing Hub. Providers continue to manage courses, pricing, and content through their existing dashboard. The Student Hub reads this data via a synchronization layer — it never writes back to the provider-side Supabase instance.

2.3 Constraints & Boundaries

Hard Constraints
  • Course data lives in Supabase — The Provider Marketing Hub stores all course metadata, content, and pricing in a Supabase PostgreSQL instance. The Student Hub must read from this source of truth. It cannot modify provider-side data.
  • Student data must NOT live in Supabase — Student profiles, enrollments, progress, and transaction records must be stored in a separate database system, fully decoupled from the provider-side infrastructure.
  • Separate auth system — Students authenticate via a dedicated identity provider, not the Supabase Auth used by providers. There is no SSO requirement between the two systems.
  • Web-only — No native mobile application. The web app must be fully responsive and performant on mobile browsers, but native apps are out of scope.
  • Stripe for payments — All payment processing routes through Stripe. No alternative payment gateways at launch.

3.2 Architectural Principles

Every decision in this architecture traces back to five core principles:

Principle What It Means How It Shows Up
Edge-First Serve as much as possible from CDN edge nodes closest to the user. Minimize round trips to origin servers. SSG for catalog pages, edge middleware for auth checks, Upstash Redis (edge-compatible), image optimization at the edge.
Read/Write Separation The read path (catalog browsing, search) and write path (enrollment, payment, progress) are architecturally distinct and scale independently. Catalog reads hit cached SSG pages + Meilisearch. Writes go through GraphQL mutations to PostgreSQL. A traffic spike on the catalog never degrades checkout availability.
Provider Isolation The Student Hub never writes to the Marketing Hub's Supabase. Data flows one direction: Provider → Student Hub via a sync service. Course Sync Service reads from Supabase (Realtime subscriptions or polling). If the Marketing Hub goes down, the Student Hub continues serving from its local catalog mirror.
Zero PCI Scope No credit card numbers, CVVs, or sensitive payment data ever touch our servers, databases, or logs. Stripe Checkout / Stripe Elements handle all card input client-side. Our backend only receives Stripe tokens and webhook events. This qualifies us for PCI SAQ-A (the lightest compliance tier).
Type-Safe Boundaries Every boundary between systems is typed: frontend ↔ API, API ↔ database, API ↔ external services. No any, no untyped JSON blobs. GraphQL with Pothos provides compile-time type safety from React component to API resolver. Drizzle ORM provides typed database queries. Zod schemas validate all external input at runtime.

Why Not SurrealDB or NATS?

Both were considered during early discovery. SurrealDB is an innovative multi-model database, but for a production system handling financial transactions at scale, PostgreSQL's 30+ years of battle-testing, ACID guarantees, mature tooling ecosystem (Drizzle, pgAdmin, connection pooling via PgBouncer/Neon), and vast hiring pool make it the lower-risk choice. SurrealDB's graph and document capabilities aren't needed for the Student Hub's predominantly relational data model (students → enrollments → courses → progress).

NATS/Jetstream excels at high-throughput, low-latency messaging between microservices. However, the Student Hub's real-time needs are limited to notifications (not chat or collaboration). For this scope, Upstash Redis pub/sub or even simple database polling is sufficient and avoids introducing an additional infrastructure component to operate, monitor, and scale. If real-time requirements expand (e.g., live sessions, collaborative features), NATS can be introduced as an additive layer without rearchitecting.

Operational Simplicity

Every component in the deployment topology is a managed service. There are no VMs to patch, no containers to orchestrate, no databases to back up manually. This is a deliberate trade-off: slightly higher per-request cost at scale in exchange for zero ops overhead at launch. As the platform grows, specific components (e.g., the database) can be migrated to self-managed infrastructure if cost optimization demands it — but starting managed means the team ships features, not infrastructure.

04 Frontend Architecture

The frontend is the student's entire experience. Every decision here — framework, rendering strategy, component design, state management — directly impacts the metrics that matter most: time to first meaningful paint, interaction responsiveness, and conversion rate on course purchases. This section details the architecture from framework selection down to the component tree.

4.1 Framework: Next.js 16 + React 19

Next.js 16 with the App Router is the foundation of the Student Hub frontend. This is not a default choice — it's a deliberate selection driven by four requirements that eliminate most alternatives:

Requirement Why It Matters How Next.js 16 Delivers
SEO for course pages Organic search is a primary acquisition channel; course detail pages must be fully indexable Server-side rendering (SSR) and static generation (SSG) produce complete HTML at the edge
Sub-second catalog loads Every 100ms of added latency reduces conversion by ~1%; the catalog is the top-of-funnel ISR (Incremental Static Regeneration) serves stale-while-revalidate cached pages from the CDN
Rich interactive player The content player (quizzes, code playgrounds, exercises) demands client-side interactivity React Server Components stream the shell; client components hydrate only interactive islands

Alternatives Considered

Remix: Excellent data-loading primitives, but weaker ISR story and smaller ecosystem for the component libraries we need (shadcn/ui is Next.js-native). Remix's loader pattern is elegant but doesn't offer the same edge-caching granularity as Next.js ISR.

Astro: Best-in-class for content sites, but its island architecture adds friction for a highly interactive app like the course player. The Student Hub isn't a blog — it's a web application with document-like pages.

SPA (Vite + React Router): Simplest DX, but sacrifices SSR/SSG entirely. Course pages would be invisible to search engines without a separate prerendering solution, and initial load times on mobile would suffer from full client-side rendering.

The "Use Client" Budget

We enforce a simple rule: every 'use client' directive must be justified in a code comment explaining which browser API or interactive behavior requires it. Why Not Material UI or Chakra?

Both are excellent libraries, but they ship as opinionated npm packages with their own styling engines. The trade-off is slightly more initial setup, which shadcn's CLI handles.

Why Zustand Over Redux, Jotai, or Context?

Redux: Powerful but overkill — the Student Hub has very little global client state (most state is server state managed by TanStack Query). Redux's boilerplate and middleware system add complexity without proportional benefit. Jotai: Atomic model is elegant but less intuitive for the team's existing experience with flux-like patterns. Zustand offers a single-store model that's easier to debug and reason about. React Context: Fine for truly static values (theme, locale), but causes unnecessary re-renders when used for frequently-changing state. Zustand's selector-based subscriptions solve this natively. Zustand is ~1KB gzipped, has zero boilerplate, and the entire store definition fits in a single file. For the Student Hub's minimal client state needs, this is the right size.

05 API Layer — GraphQL (Yoga + Pothos)

The Student Hub exposes a GraphQL API as its primary data layer — a standards-based, schema-driven interface that serves the Next.js frontend today and is designed to serve mobile apps, partner integrations, and other external consumers tomorrow. The stack is built on GraphQL Yoga (a lightweight, serverless-native GraphQL server by The Guild) with Pothos (a code-first, TypeScript-native schema builder), giving us the multi-client reach of GraphQL with the type safety of a code-first approach — no SDL files to maintain, no runtime type mismatches.

5.1 Why GraphQL

The Student Hub API is not a private BFF — it's a platform API designed for multi-client consumption from day one. The roadmap includes mobile apps, LMS integrations, and partner-facing endpoints. This forward-looking requirement drives the technology choice:

5.5 DataLoader & N+1 Prevention

GraphQL's nested resolution model makes it susceptible to the N+1 query problem — fetching a list of 20 courses and then their providers would naively execute 21 database queries. The Student Hub uses DataLoader (request-scoped, per-field batching) to collapse these into 2 queries. 

  N+1 Prevention — DataLoader in Action
  ───────────────────────────────────────

  Query:
    courses(first: 20) {
      title
      provider { name avatarUrl }    ← N+1 risk here
      reviews(first: 3) { rating }   ← N+1 risk here too
    }

  Without DataLoader (21 + 20 = 41 queries):
  ────────────────────────────────────────────
  SELECT * FROM courses LIMIT 20;                           -- 1 query
  SELECT * FROM providers WHERE id = $1;  -- repeated 20x  -- 20 queries
  SELECT * FROM reviews WHERE course_id = $1 LIMIT 3;      -- 20 queries
                                                            ──────────
                                                            41 queries 🔴

  With DataLoader (3 queries):
  ────────────────────────────
  SELECT * FROM courses LIMIT 20;                           -- 1 query
  SELECT * FROM providers WHERE id IN ($1..$20);            -- 1 batched query
  SELECT * FROM reviews WHERE course_id IN ($1..$20)        -- 1 batched query
    AND rank <= 3;  (window function)
                                                            ──────────
                                                            3 queries  🟢

Figure 5.4 — DataLoader batching: N+1 queries collapsed to constant-count queries via request-scoped batching

DataLoaders are created fresh per request (preventing cross-request cache pollution) and registered on the GraphQL context. Pothos's Drizzle plugin auto-generates loaders for common patterns (foreign key lookups), while custom loaders handle complex cases like windowed review fetching.

Each route segment in the App Router has an error.tsx boundary that catches GraphQL errors thrown during SSR. Why Not Microservices?

These "services" are module boundaries, not separately deployed microservices. They live in the same codebase, deploy in the same Vercel project, and call each other via TypeScript imports — not HTTP. This gives the organizational benefits of service-oriented architecture (clear ownership, testability, separation of concerns) without the operational cost of distributed systems (service discovery, network latency between services, distributed tracing complexity). If a service needs independent scaling in the future, it can be extracted into a standalone Vercel function or a separate deployment — but premature extraction would add complexity with no current benefit.

Idempotency Is Non-Negotiable

Stripe may deliver the same webhook event multiple times (at-least-once delivery). Without idempotency, a student could be enrolled twice or charged twice. The payment_intent.id serves as the idempotency key: before processing, the webhook handler checks if a transaction with that ID already exists. If it does, the handler returns 200 OK immediately — Stripe stops retrying, and no duplicate work occurs. This check-then-insert is wrapped in a serializable database transaction to prevent race conditions from concurrent webhook deliveries.

Why Meilisearch Over Algolia or Elasticsearch?

Algolia: Best-in-class hosted search, but pricing scales per search operation — at high traffic volumes, costs become significant. Meilisearch Cloud offers comparable sub-50ms performance at a fraction of the cost, with the option to self-host if needed.

Elasticsearch: Extremely powerful but operationally heavy — requires cluster management, shard tuning, and JVM configuration. For the Student Hub's use case (full-text search over a course catalog of 10K–100K documents), Elasticsearch is dramatically over-provisioned. Meilisearch is purpose-built for exactly this scale: instant search, faceted navigation, and typo tolerance with zero ops.

When to Extract a Service

The trigger for extracting a module into a separately deployed service is concrete, not speculative: (1) the function consistently hits Vercel's 10-second execution limit (e.g., certificate PDF generation for large courses), (2) the function needs a different scaling profile (e.g., search indexing bursts during bulk imports), or (3) a different team needs to deploy it on an independent cadence. Until one of these triggers fires, extraction is premature complexity. The modular boundaries in the current code make future extraction a straightforward refactor, not a rewrite.

08 Authentication & Identity

Authentication is handled by Clerk, a managed identity platform that provides registration, login, session management, MFA, and social OAuth — all without building or maintaining auth infrastructure. The Student Hub's auth system is completely separate from the Marketing Hub's Supabase Auth — there is no SSO between them, no shared user table, and no cross-system session. This is intentional: students and providers are different user populations with different identity requirements.

8.1 Why Clerk

Building auth from scratch (JWT issuance, password hashing, session rotation, OAuth provider integration, MFA, device management) is a significant and ongoing security liability. Clerk externalizes all of this with a developer experience optimized for Next.js.

The Marketing Hub uses Supabase Auth because providers are tightly coupled to Supabase's database and RLS policies. Using Supabase Auth for students would either (a) require the Student Hub to depend on the Marketing Hub's Supabase project (creating a dangerous coupling), or (b) require a second Supabase project just for auth (unnecessary complexity when Clerk does it better for Next.js). Clerk provides cleaner separation, better edge middleware support, and a richer component library.

Defense in Depth

Authentication is enforced at three layers: (1) Edge middleware blocks unauthenticated access to protected pages (instant redirect). (2) Pothos auth scopes reject unauthenticated GraphQL queries/mutations at the resolver level (returns GraphQL error). (3) Service-layer ownership checks ensure students can only access their own data (WHERE student_id = ctx.userId). Bypassing one layer doesn't bypass the others — a compromised edge cache, for example, cannot leak authenticated data because the resolver independently verifies identity.

    checkout:     new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(5, '60s') }),
  };

  // Yoga plugin — runs before resolver execution
  function useRateLimit(): Plugin {
    return {
      onExecute({ args }) {
        const ctx = args.contextValue;
        const isMutation = args.document.definitions.some(
          (d) => d.kind === 'OperationDefinition' && d.operation === 'mutation'
        );

        const limiter = isMutation
          ? rateLimiters.mutation
          : ctx.userId
            ? rateLimiters.authQuery
            : rateLimiters.unauthQuery;

        const key = ctx.userId ?? ctx.ip;
        const { success, remaining, reset } = await limiter.limit(key);

        if (!success) {
          throw new GraphQLError('Rate limit exceeded', {
            extensions: {
              code: 'RATE_LIMITED',
              retryAfter: Math.ceil((reset - Date.now()) / 1000),
            },
          });
        }
      },
    };
  }

    429 Too Many Requests

    X-RateLimit-Remaining: 0

9.7 Webhook Security

Webhooks are the primary ingress for external events (Stripe payments, Clerk user lifecycle, Marketing Hub content sync). Each webhook source uses cryptographic signature verification to prevent spoofing.

Source Signature Method Verification Replay Protection
Stripe HMAC-SHA256 via Stripe-Signature header stripe.webhooks.constructEvent(body, sig, secret) Timestamp tolerance: 300s (5 min). Events older than 5 min are rejected
Clerk Svix signatures (HMAC-SHA256) via svix-id, svix-timestamp, svix-signature headers svix.verify(body, headers) Message ID deduplication + timestamp tolerance. Svix retries with exponential backoff for 3 days
Marketing Hub HMAC-SHA256 with shared secret Custom verification: crypto.timingSafeEqual(computedSig, receivedSig) Timestamp in payload, reject if >5 min old. Nonce-based deduplication via Redis
Timing-Safe Comparison

All signature comparisons use crypto.timingSafeEqual() (or the equivalent in the Stripe/Svix SDKs) to prevent timing attacks. A naive === comparison leaks information about the expected signature through response time differences, allowing an attacker to brute-force the signature byte by byte.

Security Philosophy

The Student Hub's security posture is built on three principles: (1) Externalize sensitive operations — auth lives in Clerk, payments live in Stripe, neither credentials nor card data ever touches our database. 10.4 Database Migrations

Database migrations use Drizzle Kit, which generates SQL migrations from TypeScript schema changes. The migration strategy differs by environment to balance safety with developer velocity.

Environment Migration Strategy Rollback

Production drizzle-kit migrate — runs in Vercel build step before deployment Neon point-in-time restore + Vercel deploy rollback. Migration rollback scripts generated but only applied manually
Migration Safety

Production migrations are applied before the new code deploys. This means the database schema is always forward-compatible with both the old and new application versions during the deployment window. Destructive migrations (dropping columns/tables) are split into two deploys: (1) deploy code that stops reading the column, (2) deploy migration that drops the column. This prevents errors during the brief overlap when old code is still running.

11 Scalability & Performance

The Student Hub is designed to launch at 10K monthly active students and scale to 100K+ without architectural changes — only configuration tuning and tier upgrades. Beyond 100K, specific components may need to be re-evaluated, but the core architecture remains sound. This section documents the scaling characteristics of each layer, the bottlenecks that will appear first, and the mitigation strategies for each.

11.4 Database Connection Pooling

Serverless functions create a new database connection on every cold start. Without pooling, this quickly exhausts PostgreSQL's connection limit (default: 100). Neon provides a built-in connection pooler (PgBouncer-compatible) that solves this.

  │  #1      │  │  #2      │  │  #50     │     = 50 direct connections
  └────┬─────┘  └────┬─────┘  └────┬─────┘     → PostgreSQL overwhelmed
       │              │              │
       └──────────────┼──────────────┘
                      ▼
  ┌────────────────────────────┐
  │  Neon PostgreSQL           │
  │  max_connections = 100     │
  │  50/100 used = 50% 😰     │
  └────────────────────────────┘


  With Neon pooler (solution):
  ┌──────────┐  ┌──────────┐  ┌──────────┐

11.6 Bottleneck Playbook

When scaling limits are reached, the following playbook provides the mitigation strategy for each bottleneck, in order of expected occurrence.

# Bottleneck Symptom Trigger (~MAU) Mitigation
1 Neon compute Query latency increase; p95 >500ms ~50K Increase Neon compute units (0.25 → 1 → 4 CU). Auto-scaling handles spikes. ~2 min, no downtime
2 Serverless concurrency 429 errors from Vercel; cold start rate increases ~80K Increase Vercel function concurrency limit. Consider edge runtime for GraphQL (no cold start). Enable Vercel's "Fluid Compute" for long-running functions
3 Meilisearch capacity Search latency >100ms; indexing lag >10s ~100K Upgrade Meilisearch Cloud tier. If catalog exceeds 1M documents, evaluate sharding or dedicated instance
4 Database read throughput Read queries competing with writes; connection pool saturation ~200K Add Neon read replicas. Route read-heavy queries (catalog, search aggregation) to replicas. Drizzle supports multiple connection targets
5 GraphQL complexity Schema size makes query planning expensive; type generation slow ~500K Enable Automatic Persisted Queries (APQ). Consider schema federation if the API surface grows beyond one team's ownership
What Doesn't Scale (and Doesn't Need To)

Some operations are inherently low-volume and never become bottlenecks: Webhook ingestion (proportional to content changes and payments, not page views — at most ~1K/day at 100K MAU), Clerk auth (edge JWT verification has zero backend cost — scales infinitely), and Stripe checkout (~1–2% conversion rate means ~1–2K checkouts/day at 100K MAU — Stripe handles billions).

Scaling Philosophy

The Student Hub's scaling approach is "optimize reads, accept writes". Read traffic (catalog, search) is 50–100× higher than write traffic (enrollment, progress) and is fully absorable by caching layers. Write traffic at marketplace scale is surprisingly modest — even at 100K MAU, the database sees only ~3 QPS of actual write queries. The architecture's bottleneck is designed to be the database (the component with the most linear scaling options), not the application layer. When the database eventually becomes the bottleneck, Neon's autoscaling, read replicas, and connection pooling provide a graduated scaling path without application code changes.

12.3 Risk Register

The following risks are acknowledged and have explicit mitigation strategies. These are not reasons to avoid the chosen architecture — they are scenarios to monitor and respond to if they materialize.

Risk Likelihood Impact Mitigation
Vendor lock-in (Vercel) Medium Medium Next.js is open-source and runs on any Node.js host. ISR can be replicated with CDN + cache headers. Migration cost is moderate (reimplement edge middleware, ISR, preview deploys)
Neon outage Low High Neon provides 99.95% SLA. Read traffic survives via cache layers (ISR + Redis). Write traffic (enrollments, payments) would be temporarily unavailable — Stripe webhooks retry for 72h, so no data loss
Clerk pricing at scale Medium Low At 1M+ MAU, Clerk's per-MAU pricing may become significant. Migration to self-hosted auth (using existing Clerk user export) is feasible but costly in engineering time. Monitor cost/MAU ratio quarterly
Catalog sync lag Low Low Webhook-based sync typically completes in <5s. If Marketing Hub webhook delivery fails, manual resync endpoint exists. Students see "last updated" timestamp on course pages
GraphQL schema evolution Medium Medium Pothos makes schema changes type-safe (breaking changes fail compilation). CI checks for schema breaking changes. Client uses document-based caching (no normalized cache to invalidate). Deprecation-first approach for field removal
Cold start latency spike Medium Low Serverless cold starts (~200ms) only affect cache misses on non-ISR pages. ISR eliminates cold starts for 90% of traffic. Vercel's "Fluid Compute" feature can pre-warm functions if needed
Meilisearch index corruption Low Medium Full reindex from PostgreSQL catalog_courses table. Reindex script runs in ~30s for 100K documents. Search degradation doesn't affect purchases or enrollment
